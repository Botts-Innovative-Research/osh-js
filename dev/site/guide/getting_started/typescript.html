# Typescript

It is possible to use the Toolkit within a Typescript application. For example, by using Webpack and some additional loaders.

# Dependencies

To configure Typescript support, we will need the following dependencies:

$ yarn add typescript
$ yarn add -D @babel/core @babel/preset-env @babel/preset-react @babel/preset-typescript babel-loader loader-utils ts-loader webpack webpack-cli webpack-dev-server

If you use React:

$ yarn add -D @babel/preset-react @types/react @types/react-dom
$ yarn add react react-dom react-scripts

# Webpack > 5.x setup

For webpack > 5.0, it is necessary to specify some parameters for this to work. First, we have to manage the case of crypto/path/fs that will prevent Webpack to load correctly the configuration when the 'target' is not nodejs. For that, we will add in fallback these elements into webpack.config.js:

    fallback: {
      "crypto" : false,
      "path": false,
    }

We need then to set up a project and install some dependencies. For Webpack, we need webpack and webpack-cli package and for TypeScript features, we need typescript and ts-loader packages.

$ yarn add -D webpack webpack-cli typescript ts-loader

The most important part of this configuration is module.rules. This is a list of loader configuration rules that will be matched for all entry files (including imported files). If the path of the file matches the test pattern but not the exclude pattern, then it will be filtered through loaders specified by use:

      {
        test: /\.tsx?$/,
        use: 'ts-loader',
        exclude: /node_modules/
      }

The resolve.extensions list contains the extension Webpack will use to search a file when an import statement is found in a program. For example, import x from './y' will resolve in the lookup of ./y.ts and ./y.js according to the above configuration. Therefore, you don’t need to provide all the program files as entry files in the Webpack configuration.

 extensions: ['.tsx', '.ts', '.js']

That's about it for webpack, here is a full working example of a webpack configuration:

// webpack.config.common.js
const CopywebpackPlugin = require('copy-webpack-plugin');
const { DefinePlugin, ProvidePlugin } = require('webpack');
const HtmlWebpackPlugin = require('html-webpack-plugin');
const { CleanWebpackPlugin } = require('clean-webpack-plugin');
// Common configs
const path = require('path');

const cesiumConfig = {
  amd: {
    // Enable webpack-friendly use of require in Cesium
    toUrlUndefined: true
  },
  plugins: [
    // Copy Cesium Assets, Widgets, and Workers to a static directory
    new CopywebpackPlugin({
          patterns: [
            {from: path.resolve(__dirname, 'node_modules/cesium/Source/Workers'), to: 'Workers'},
            {from: path.resolve(__dirname, 'node_modules/cesium/Source/Assets'), to: 'Assets'},
            {from: path.resolve(__dirname, 'node_modules/cesium/Source/Widgets'), to: 'Widgets'},
            {from: 'images', to: 'images'},
            {from: 'models', to: 'models'}
          ],
          options: {
            concurrency: 100,
          }
    }),
  ],
};
// Now, using the cesiumConfig in your real configuration
const config = {
  ...cesiumConfig,
  entry: "./src/index.tsx",
  output: { path: path.join(__dirname, "build"), filename: "index.bundle.js" },
  mode: process.env.NODE_ENV || "development",
  resolve: {
    alias: {
      'osh-js': path.resolve(__dirname, '../../../source'),
      'cesium': path.resolve(__dirname, 'node_modules/cesium'),
    },
    extensions: ['.tsx', '.ts', '.js'],
    fallback: {
      "crypto" : false,
      "path": false,
      "fs": false
    }
  },
  devServer: { contentBase: path.join(__dirname, "src") },
  module: {
    unknownContextCritical: false,
    rules: [
      {
        test: /\.(js|jsx)$/,
        exclude: /node_modules/,
        use: {
          loader: "babel-loader"
        }
      },
      {
        test: /\.tsx?$/,
        use: 'ts-loader',
        exclude: /node_modules/,
      },
      {
        test: /\.(png|svg|jpg|jpeg|gif|ico)$/,
        use: [
          'file-loader',
        ],
      },{
        test: /\.css$/i,
        use: ['style-loader', 'css-loader'],
      },{
        test: /\.worker\.js$/,
        use: { loader: 'worker-loader', options: { filename: 'Worker.[chunkhash].js' } }
      }
    ],
  },
  stats: {
    children: true,
  },
  optimization: {
    // We no not want to minimize our code.
    minimize: false
  },
  plugins: [
  ...cesiumConfig.plugins,
    new CleanWebpackPlugin(),
    new HtmlWebpackPlugin({
      template: 'public/index.html',
      filename: './index.html',
      favicon: './public/favicon.ico'
    }),
],
};

module.exports = config;

# Typescript setup

The second important part of the project compilation is the tsconfig.json file that controls how a TypeScript file is compiled. At the moment, a .tsx file will be transpiled to the ES6 version of JavaScript (target) and all import statements will be kept in the ES6 format (module) in the output.

The noImplicitAny compiler-option is used to disallow a value implicitly having the any type. If any error TypeScript related error occurs during the compilation (such as implicit any), Webpack will stop with an error by printing the TSC errors to the console.

For a good integration with the Toolkit and its ES6 modules, several parameters must be defined correctly:

    "target": "es6",
    ...
    "paths": {
        "cesium": ["node_modules/cesium/Build/Cesium/Cesium.js"],
        "osh-js/*": ["../../../source/*"]
    },
    ...
    "baseUrl": ".",
    "allowJs": true,
    "strict": false,
    "isolatedModules": true,
    "noEmit": false,
    "noImplicitAny": false,
    ...

Defining paths allows you to create an alias to an external source code, for example, our Toolkit source code. This works in the same way as in Webpack's resolve. It is important to define it in both places

That's about it for tsconfig.json, here is a full working example of a tsconfig configuration:

{
  "compilerOptions": {
    "target": "es6",
    "lib": [
      "dom",
      "dom.iterable",
      "esnext"
    ],
    "paths": {
      "cesium": ["node_modules/cesium/Build/Cesium/Cesium.js"],
      "osh-js/*": ["../../../source/*"]
    },
    "baseUrl": ".",
    "allowJs": true,
    "skipLibCheck": true,
    "esModuleInterop": true,
    "allowSyntheticDefaultImports": true,
    "strict": false,
    "forceConsistentCasingInFileNames": true,
    "noFallthroughCasesInSwitch": true,
    "module": "es6",
    "noImplicitAny": false,
    "moduleResolution": "node",
    "resolveJsonModule": true,
    "isolatedModules": true,
    "noEmit": false,
    "jsx": "react-jsx",
    "declaration": true
  },
  "include": [
    "src"
  ]
}

# Babel

The last part concerns babel which allows to transpile TS + JS coding in the same target. In our example, we will use the babel presets and we will configure a babel specific configuration file (.babelrc):

{
  "presets": [
    "@babel/env",
    "@babel/react",
    "@babel/preset-typescript"
  ],
  "plugins": [
    "@babel/plugin-proposal-class-properties"
  ]
}
